# Plan 02-03 — ROCm-on-Windows spike (documentation only, no code)

## Phase Goal

**As a** user on the desktop, **I want to** know whether my 6800 XT actually runs the TheRock nightly ROCm path, **so that** Phase 3 picks the right STT engine on the first try instead of guessing.

## Objective

Run an empirical spike on the actual desktop box (RDNA2 / gfx1030 / RX 6800 XT) to determine whether the TheRock nightly `gfx103X-dgpu` wheel path works in mid-2026, and to document the working path or the documented fallback in `02-03-SPIKE.md`. This is a **documentation-only plan** — there is no production code in this plan. The deliverable is a single markdown file that Phase 3 (STT adapter) will read before picking the inference engine. The accompanying `tests/test_spike_documented.py` is a **contract guard** that asserts the file exists with the five required sections; it does not re-run the spike. Per D-07, Phase 2's detect code (02-01) ships targeting the documented paths regardless of the spike verdict — the fallback chain (CPU STT/LLM, `n_gpu_layers=0`, loud log) already handles failure safely, so shipping unverified is acceptable. The spike is the load-bearing verification flagged in STATE.md "Blockers/Concerns."

**Purpose:** The 02-03 spike is the empirical verification of the desktop ROCm path. The verdict is consumed by Phase 3 to decide the STT engine (faster-whisper on CPU vs whisper.cpp ROCm fork vs TheRock torch). Phase 2 code does not depend on the verdict; the spike refines the documented path for the next phase.

**Output:**
- `.planning/phases/02-gpu-backend-detection-model-manager/02-03-SPIKE.md` — the spike deliverable (the only artifact of this plan that the user writes by hand)
- `tests/test_spike_documented.py` — a single test file with four test functions that act as the contract guard

```python
"""Contract guard for 02-03-SPIKE.md.

This test exists to ensure the spike deliverable is present and well-formed
before Phase 3 plans can be written. The spike is the empirical evidence that
the desktop ROCm path works (or that the fallback is documented per D-07); the
test asserts the file's structure, not its content correctness.

If this test fails, the spike has not been written yet OR the verdict heading
is missing OR one of the required sections is missing. Fix: write the spike.
"""
from pathlib import Path
import pytest

SPIKE_PATH = Path(__file__).parent.parent / ".planning" / "phases" / "02-gpu-backend-detection-model-manager" / "02-03-SPIKE.md"

REQUIRED_SECTIONS = [
    "## 1. Target environment",
    "## 2. What worked",
    "## 3. Fallback decision",
    "## 4. Pitfalls hit",
    "## 5. What Phase 3 must do",
]

VALID_VERDICTS = (
    "VERDICT: ROCM_VIA_THEROCK_WORKS",
    "VERDICT: ROCM_FALLBACK_TO_CPU",
)


def test_spike_file_exists():
    assert SPIKE_PATH.exists(), f"Spike file not found: {SPIKE_PATH}"


def test_spike_file_has_required_sections():
    body = SPIKE_PATH.read_text(encoding="utf-8")
    for section in REQUIRED_SECTIONS:
        assert section in body, f"Missing required section: {section}"


def test_spike_file_has_valid_verdict():
    body = SPIKE_PATH.read_text(encoding="utf-8")
    assert any(verdict in body for verdict in VALID_VERDICTS), (
        f"Missing or invalid verdict. Must be one of: {VALID_VERDICTS}"
    )


def test_phase3_section_has_at_least_one_requirement():
    body = SPIKE_PATH.read_text(encoding="utf-8")
    section_start = body.index("## 5. What Phase 3 must do")
    section_body = body[section_start:]
    next_section_match = section_body.find("\n## ", 1)
    if next_section_match > 0:
        section_body = section_body[:next_section_match]
    assert "must" in section_body.lower(), (
        "Phase 3 section must contain at least one 'must' requirement"
    )
```
