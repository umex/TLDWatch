# Plan 02-01 â€” First-run GPU detect + burn-in test + settings.json wire-in

## Objective

Silently detect the active GPU backend (CUDA / ROCm / CPU) on first boot, run a real-kernel burn test to confirm the path actually uses the GPU (not silent CPU fallback, Pitfall 1 / 12), and persist the result to `data/settings.json` atomically. Extend `Settings` and `UpdateSettingsRequest` with the seven Phase 2 fields so the on-disk schema carries everything the model manager and the future settings panel will read. Add three back-end endpoints (`POST /diagnostics/gpu-burn`, `GET /diagnostics/vram`, `POST /diagnostics/test-hf-token`) that the future settings panel uses.

**Purpose:** The user must never have to know which GPU is in the box. SC-1 ("First run silently writes settings.json with the right backend") is fully delivered here. SC-2's preset table is partially declared here (the enum + ModelSpec + ModelSet + PRESETS dict live in 02-02; 02-01 only declares the `quality_preset: QualityPreset = QualityPreset.BALANCED` field and the `per_category_overrides: ModelSet | None = None` field on `Settings` â€” the values that fill them are written in 02-02's manifest work).

**Output:**
- `app/models/diagnostics.py` â€” `BackendProbe`, `VRAMState`, `LoadedModelInfo` (response models)
- `app/models/backend.py` â€” `GpuBackend` enum + `detect()` + `burn_test()` functions
- `app/models/vram.py` â€” `probe_vram()` function
- `app/models/hf_token.py` â€” `validate_token()` function (HF Hub HEAD shim)
- Extended `app/models/settings.py` with 7 new fields and the strict `UpdateSettingsRequest` extension
- `app/api/routes_diagnostics.py` â€” three new routes
- Extended `app/main.py` lifespan: detect on first boot, write settings.json, register the new router, register new Pydantic models in `_EXTRA_OPENAPI_MODELS`
- Extended `pyproject.toml` with `huggingface_hub>=0.25`, `psutil>=5.9`; dev deps already cover pytest + httpx
- 4 new test files + extended `tests/conftest.py` with a `mock_hf_hub_url` fixture (Phase 2 mock seam)

### New module `app/models/backend.py`
- Module-level `_log = logging.getLogger(__name__)` (the `app.jobs.service.py:37` pattern).
- `async def detect() -> GpuBackend` â€” Stage 1 cheap probe. Order: (1) `subprocess.run(["pip", "show", "torch"], timeout=3, capture_output=True, text=True)` â€” parse the `Version:` line for `+cu` or `+rocm` substring; (2) `subprocess.run(["nvidia-smi", "--query-gpu=name,memory.total", "--format=csv"], timeout=3, ...)`; (3) `os.environ.get("ROCM_PATH") or os.environ.get("HIP_PATH")`; (4) `import torch; torch.cuda.is_available()` and `torch.cuda.device_count()` and `torch.version.hip`; (5) `subprocess.run(["powershell", "-Command", "Get-WmiObject Win32_VideoController | Select-Object Name"], timeout=3, ...)` for the device name. Returns `GpuBackend.CUDA` if step (1) shows `+cu` AND step (2) parses AND step (4) reports a device; returns `GpuBackend.ROCM` if step (1) shows `+rocm` AND step (4) reports a device (HIP is `torch.cuda` API); returns `GpuBackend.CPU` otherwise. All `subprocess.run` calls are wrapped in `try/except (subprocess.TimeoutExpired, FileNotFoundError)` â€” a missing `nvidia-smi` is not an error, it's just an empty result.
- `async def burn_test(backend: GpuBackend) -> BackendProbe` â€” Stage 2 real-kernel run. Per backend:
  - `CUDA` / `ROCM` (HIP): `import torch; a = torch.randn(1024, 1024, device="cuda"); b = torch.randn(1024, 1024, device="cuda"); torch.cuda.synchronize(); t0 = time.perf_counter(); (a @ b).item(); torch.cuda.synchronize(); t1 = time.perf_counter()`; record `burn_test_ms = (t1 - t0) * 1000`. Read `device_name = torch.cuda.get_device_name(0)` and `vram_total_mb = int(torch.cuda.mem_get_info(0)[1] / 1024**2)`. If `burn_test_ms > 5000`, log a warning ("kernel took >5s; falling back to CPU"); the probe is still returned (the fallback to CPU is the user's call, not ours).
  - `CPU`: `device_name = "CPU"`, `driver_version = None`, `vram_total_mb = None`, `burn_test_ms = None`, `notes = "no GPU detected; running in CPU mode"`.
  The function returns a complete `BackendProbe` with `probed_at = utcnow_iso()` and `notes` filled from any warnings.
- The function is `async def` (mirroring the `app.jobs.reconcile_all` async signature) so the lifespan can `await` it. Inside, the torch calls are synchronous (no real async API in torch); wrap the synchronous block in `asyncio.to_thread(...)` only if a real I/O call is added (Phase 2 keeps it sync-in-async for simplicity, matching `app.jobs.manifest.write_manifest`'s pattern).

### New module `app/models/vram.py`
- `def probe_vram(backend: GpuBackend, manager_state: ManagerState) -> VRAMState` â€” `ManagerState` is a typed `BaseModel` in this module with one field: `live_vram_bytes: dict[ModelCategory, int] = Field(default_factory=dict)`. The probe is sync (the torch call is fast â€” no need to async-wrap). Per backend:
  - `CUDA` / `ROCM`: `if not torch.cuda.is_available(): return VRAMState(backend=backend, total_mb=0, available_mb=0, used_mb=0, loaded=[])`; `free, total = torch.cuda.mem_get_info(0)`; `torch_alloc = torch.cuda.memory_allocated(0)`; `llm_pool = sum(manager_state.live_vram_bytes.values())`; `used = torch_alloc + llm_pool`; `available = free - llm_pool`; return a `VRAMState` with `total_mb = int(total / 1024**2)`, `available_mb = int(available / 1024**2)`, `used_mb = int(used / 1024**2)`, `loaded` populated from the `ManagerState` (a list of `LoadedModelInfo` constructed from the dict).
  - `CPU`: `ps = psutil.virtual_memory()`; return a `VRAMState` with `total_mb = int(ps.total / 1024**2)`, `available_mb = int(ps.available / 1024**2)`, `used_mb = int(psutil.Process(os.getpid()).memory_info().rss / 1024**2)`, `loaded = []`.
- The function is import-safe: `import torch` and `import psutil` are inside the function body (NOT at module top), so a test that doesn't import torch (e.g., CPU-only) does not crash. This is the same lazy-import discipline `app.storage.db` uses for the engine listener.
- `VRAMState.loaded` is computed in the function (the manager state is the single source of truth). 02-02 wires `ModelManager` to maintain `ManagerState.live_vram_bytes` on every `load()` / `unload()`.

### New module `app/models/hf_token.py`
- `async def validate_token(token: str | None, repo_id: str = "pyannote/speaker-diarization-3.1") -> HfTokenResult` â€” module-level helper. If `token is None`, return `HfTokenResult(status="skipped", reason="no token configured")`. Otherwise build the URL via `huggingface_hub.hf_hub_url(repo_id, filename="config.yaml")` and do `httpx.AsyncClient().head(url, headers={"Authorization": f"Bearer {token}"}, timeout=5.0)`. Map the response:
  - 200 â†’ `HfTokenResult(status="ok", user=parse_user_from_headers_or_default(token))`. The user field is the HF username from the response `x-repo-author` header if present, else `None` (the front-end only needs to know "ok").
  - 401 â†’ `HfTokenResult(status="rejected", reason="token invalid")`.
  - 403 â†’ `HfTokenResult(status="rejected", reason="model terms not accepted", fix=f"visit https://huggingface.co/{repo_id}")`.
  - any other code (e.g. 5xx, timeout) â†’ `HfTokenResult(status="skipped", reason=f"upstream error {code}")` (the app does not refuse to start on a flaky HF check; Pitfall 3).
- Lazy `import httpx` and `import huggingface_hub` inside the function body so the module is import-safe in a test that doesn't have `huggingface_hub` installed yet (the dependency is added in this plan).
