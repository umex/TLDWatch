# Plan 01-01 — Walking Skeleton: FastAPI boot + first end-to-end job creation

## Objective

Scaffold the back-end service and ship the thinnest end-to-end working slice. After this plan:
- `uvicorn app.main:app` boots the FastAPI app on localhost with CORS and trusted-host middleware configured
- The DB file is created in WAL mode with a `jobs` table and a `settings` table (one migration applied); the WAL pragma is asserted on every connection (not just the first)
- The bootstrap `settings.json` is loaded from a STABLE path next to the backend executable (not inside the relocatable `data_dir`), eliminating the data_dir circular bootstrap
- `POST /jobs` creates a job end-to-end: generates a UUIDv4, INSERTs a row in `jobs`, creates `data/jobs/<id>/`, writes a minimal `manifest.json` atomically, and returns the typed job
- `GET /openapi.json` returns a schema the front-end can later consume via `openapi-typescript`
- `GET /health` returns 200
- The four-boundary layout exists: `app.api`, `app.jobs`, `app.storage`, `app.models` — no other code imports the storage atomic-write helper or the DB session directly except through these modules
- A retry helper `app/storage/retry.py::retry_windows` exists and is wired into `atomic_write_*` to handle Windows `PermissionError` from antivirus / Search Indexer holding the file
- A timezone-aware UTC helper `app/util/time.py::utcnow_iso` is used by every timestamp the app writes

This is the architectural spine. Every later phase imports these modules without changing their public contracts.
