# Plan 01-01 — Walking Skeleton: FastAPI boot + first end-to-end job creation

## Objective

Scaffold the back-end service and ship the thinnest end-to-end working slice. After this plan:
- `uvicorn app.main:app` boots the FastAPI app on localhost
- The DB file is created in WAL mode with a `jobs` table and a `settings` table (one migration applied)
- `POST /jobs` creates a job end-to-end: generates a UUIDv4, INSERTs a row in `jobs`, creates `data/jobs/<id>/`, writes a minimal `manifest.json` atomically, and returns the typed job
- `GET /openapi.json` returns a schema the front-end can later consume via `openapi-typescript`
- `GET /health` returns 200
- The four-boundary layout exists: `app.api`, `app.jobs`, `app.storage`, `app.models` — no other code imports the storage atomic-write helper or the DB session directly except through these modules

This is the architectural spine. Every later phase imports these modules without changing their public contracts.

## must_haves (goal-backward verification)

- [ ] **truth: FastAPI app boots locally and serves OpenAPI** — `uvicorn app.main:app` starts; `GET /openapi.json` returns 200 with `paths` containing `/health` and `/jobs`
- [ ] **truth: SQLite WAL with versioned schema** — after boot, `data/app.db` exists; `SELECT * FROM schema_version` returns at least one row naming `0001_initial.sql`; `PRAGMA journal_mode = WAL` is in effect
- [ ] **truth: `POST /jobs` creates a per-job folder with a manifest** — request body `{}` returns 201 with a JSON `JobResponse` containing a UUIDv4 `id` and `status="queued"`; afterwards `data/jobs/<id>/manifest.json` exists and is valid JSON matching the Pydantic `JobManifest` model
- [ ] **truth: Pydantic models are shared with the OpenAPI schema (D-16 codegen contract)** — `GET /openapi.json` shows the `JobManifest` and `JobResponse` schemas under `components.schemas` with the exact field names `job_id`, `schema_version`, `current_stage`, `stage_timestamps`, `status`, `error`, `summary_kinds`, `source_type`, `source_path`, `source_sha256`, `duration_s`, `language`. The back-end does NOT need to call `openapi-typescript` itself — D-16 says types-only codegen is the front-end's job (Phase 5); Phase 1's only obligation is that the live `/openapi.json` is complete and `openapi-typescript`-consumable (no missing `$ref`s, every Pydantic field is present, every request/response model is registered via FastAPI's `response_model=`).
- [ ] **truth: clean four-module boundary** — `grep -rE "from app\\.storage\\.atomic|from app\\.storage\\.fs" app/ tests/` returns hits only inside `app/storage/` and `app/jobs/` (the two modules allowed to call `storage.atomic_write_*`); `app/api/*` imports the storage module only through `app.jobs.service`
- [ ] **truth: settings Pydantic model is the source of truth** — `app/models/settings.py` defines `class Settings(BaseModel): data_dir: str`; on first boot a `data/settings.json` is created with `{"data_dir": "data"}` and is loaded into the `Settings` Pydantic model (validated, not raw JSON)
- [ ] **verification**: `pytest -q` passes; `uvicorn app.main:app` boots cleanly with no errors; `curl -sf http://127.0.0.1:8000/openapi.json | python -c "import sys,json; d=json.load(sys.stdin); assert '/jobs' in d['paths']; print('ok')"` prints `ok`
