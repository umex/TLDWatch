# Plan 01-02 — Full Pydantic model surface + read endpoints + settings.json Pydantic model

## Objective

Build out the typed surface that every later phase imports. After this plan:
- All Pydantic models Phase 1 needs are in place: `TranscriptSegment`, `Transcript`, `Summary` (with `SummaryKind` literal), `JobStatus` already in place from 01-01, `Settings` with the validated `data_dir` field, and the existing `JobManifest` and `JobResponse`
- `GET /jobs` lists jobs ordered by `created_at DESC`, with optional `?status=` filter
- `GET /jobs/{id}` returns one `JobResponse` (404 if not found)
- `GET /settings` returns the current `Settings` (lax output, default config)
- `PATCH /settings` accepts a strict `UpdateSettingsRequest` and atomically rewrites `data/settings.json`
- Migration `0002_remaining_jobs_columns.sql` adds the missing `jobs` columns needed by later phases (`source_sha256`, `duration_s`, `language`, `summary_kinds_json`, `updated_at`) — fields that are part of the locked manifest schema but the DB row should also carry, so a `GET /jobs/{id}` can return the full picture without reading the manifest file
- The Pydantic `Settings` model is the source of truth: the file is its serialized form, all writes go through the atomic helper

This plan does NOT yet add `transcript_segments`, `summary_outputs`, `job_events`, `speaker_aliases`, or any other table beyond `jobs` + `settings` + `schema_version` (D-09). Those arrive when their owning phase lands.

## must_haves (goal-backward verification)

- [ ] **truth: complete Phase 1 model surface** — `app/models/{job,manifest,settings,common,transcript,summary}.py` exist; importing `TranscriptSegment`, `Transcript`, `Summary`, `SummaryKind` from `app.models.transcript` / `app.models.summary` works; `TranscriptSegment` has fields `start_s: float`, `end_s: float`, `text: str`, `speaker: str | None = None`, `confidence: float | None = None`

### New routes
- `GET /jobs` — query params: `status: str | None = None`; returns `list[JobResponse]`, ordered `created_at DESC`, capped at 200
- `GET /jobs/{id}` — returns one `JobResponse` or 404
- `GET /settings` — returns the current `Settings` (lax output, default config)
- `PATCH /settings` — accepts `UpdateSettingsRequest` (strict), atomically writes `data/settings.json`, updates the in-memory `current_settings`, returns the new `Settings`

### Modified routes
- `POST /jobs` — no change to public surface; the `create_job` service now also populates the new `jobs` columns from the request if provided (still no-op for an empty body)

### New SQL migration
- `migrations/0002_remaining_jobs_columns.sql` — adds the columns the manifest already tracks but the `jobs` row should also carry. Use `ALTER TABLE jobs ADD COLUMN source_sha256 TEXT` etc., but guard with a check in `apply_migrations` that reads `pragma_table_info('jobs')` and skips the ALTER if the column already exists. Columns added: `source_sha256 TEXT`, `duration_s REAL`, `language TEXT`, `summary_kinds_json TEXT` (JSON-encoded list, since SQLite has no native array), `updated_at TEXT`, `stage_timestamps_json TEXT` (so the row can mirror the manifest without re-reading disk for read endpoints), `error TEXT`. Do NOT add tables for transcript_segments, job_events, queue_positions, summary_outputs, speaker_aliases (D-09).

### New service functions
- `app/jobs/service.py::list_jobs(session, status=None) -> list[JobResponse]` — SELECT * FROM jobs ORDER BY created_at DESC LIMIT 200, optional WHERE status = :status
- `app/jobs/service.py::get_job(session, job_id) -> JobResponse | None` — SELECT * FROM jobs WHERE id = :id, return None if missing
- `app/settings/__init__.py` and `app/settings/service.py` — NEW module to host settings read/write logic (boundary clean: settings is its own concern, not a job concern). Functions:
  - `def load_settings_from_disk(path: Path) -> Settings` — read JSON, validate via `Settings.model_validate_json`
  - `async def save_settings_to_disk(path: Path, settings: Settings) -> None` — atomic write via `app.storage.atomic`
  - `def current_settings() -> Settings` — returns the in-memory `Settings` set by the lifespan
  - `def update_settings(patch: UpdateSettingsRequest) -> Settings` — applies the patch, persists to disk, updates the dependency-injected value, returns the new `Settings`

### New module
- `app/settings/__init__.py` — empty
- `app/settings/service.py` — the four functions above

### New test files
- `tests/test_transcript_models.py` — round-trip `TranscriptSegment`, `Transcript` from JSON; bad types rejected
- `tests/test_summary_models.py` — round-trip `Summary` for each `SummaryKind` literal; unknown kind rejected
