# Plan 01-02 — Full Pydantic model surface + read endpoints + settings.json Pydantic model

## Objective

Build out the typed surface that every later phase imports. After this plan:
- All Pydantic models Phase 1 needs are in place: `TranscriptSegment`, `Transcript`, `Summary` (with `SummaryKind` literal), `JobStatus` already in place from 01-01, `Settings` with the validated `data_dir` field, `UpdateSettingsRequest` (strict), and the existing `JobManifest` and `JobResponse` (extended with read-side fields)
- `GET /jobs` lists jobs ordered by `created_at DESC`, with optional `?status=` filter and `?limit=` / `?offset=` pagination (limit capped at 200, default 50)
- `GET /jobs/{id}` returns one `JobResponse` (404 if not found)
- `GET /settings` returns the current `Settings` (lax output, default config)
- `PATCH /settings` accepts a strict `UpdateSettingsRequest`, atomically rewrites `data/settings.json`, and returns the new `Settings` with an `X-Restart-Required: true` header when `data_dir` is the field that changed (per Codex HIGH: runtime data_dir change requires restart; the change is persisted but the engine / session factory / settings file path are NOT hot-swapped)
- Migrations 0002..0007 each add a single column to the `jobs` table (per Codex HIGH: per-statement guards via one-column-per-file, replacing the per-file `GUARDS` approach that was rejected for partial-application fragility)
- The Pydantic `Settings` model is the source of truth: the file is its serialized form, all writes go through the atomic helper
- A startup log WARNING is emitted if `data_dir` from the on-disk file differs from the default (signals manual override)

This plan does NOT yet add `transcript_segments`, `summary_outputs`, `job_events`, `speaker_aliases`, or any other table beyond `jobs` + `settings` + `schema_version` (D-09). Those arrive when their owning phase lands.

- [ ] **truth: settings round-trip through Pydantic, restart header on data_dir change** — `GET /settings` returns `{"data_dir": "data"}`; `PATCH /settings` with body `{"data_dir": "C:/some/path"}` returns 200 with the new settings and the response header `X-Restart-Required: true`; afterwards `data/settings.json` contains the new value (written atomically); the in-memory `Settings` used by the app reflects the change for code that reads `data_dir` after the PATCH; the file was written atomically (no torn writes on crash)
