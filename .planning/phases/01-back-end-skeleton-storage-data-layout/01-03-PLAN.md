# Plan 01-03 — Per-job filesystem layout, stage files, resume rule, cleanup, startup reconciliation

## Objective

Complete the per-job filesystem story and the consistency protocol between the DB and the manifest. After this plan:
- Every stage file from D-03 has a typed path helper + atomic write helper + read helper: `transcript.json`, `diarization.json`, `summary-<kind>.json`, `edits.json`, plus the existing `manifest.json` and `source.<ext>`
- The full `JobManifest` (matching the Pydantic model from Plan 01-01) can be round-tripped to/from the manifest file with `read_manifest` and `write_manifest`
- The resume rule (D-12) is implemented as `infer_resume_point(manifest) -> StageName | None`: walks the files in the standard order, respects optional stages (`diarized` skipped if `manifest.diarization_enabled == False`; `summarized` complete iff every kind in `manifest.summary_kinds` has a `summary-<kind>.json` file; `summary_kinds == []` means `summarized` is not applicable), and treats `done` as a DERIVED terminal state (not a file-backed stage) — the resume rule returns `None` when all applicable prior stages are complete AND `manifest.current_stage == "done"`. Phase 4 (orchestrator) is the only writer that flips `current_stage` to `"done"` after the last applicable stage
- Cleanup helpers exist with the correct ordering (Codex HIGH #8): `cancel_job` does `UPDATE jobs SET status='cancelled'` FIRST and commits, THEN rmtree's the folder; if the rmtree fails, the row stays cancelled and a future operation can retry the folder cleanup (the opposite order risks leaking a cancelled row with no folder to inspect). `mark_failed` keeps the folder. `mark_stale` uses a 10-minute threshold
- `update_stage` enforces the consistency protocol (Codex HIGH #1): write the stage output file → write the manifest atomically (with `current_stage` and `stage_timestamps`) → commit the DB projection (`current_stage`, `stage_timestamps_json`, `updated_at`) last. The manifest is the source of truth (D-03); the DB row is a projection
- A startup reconciliation pass (Codex HIGH #1 followup) walks every job folder on lifespan startup, re-derives `current_stage` and `stage_timestamps` from the manifest, and UPDATEs any drifted DB row. This self-heals the rare crash window between manifest write and DB commit
- A typed `ManifestPatch` model (Codex HIGH #7) replaces the loose `dict[str, Any]`; the patch only forwards allowlisted user-mutable fields, and `current_stage`, `job_id`, `schema_version`, `stage_timestamps`, `status`, `error` are PROTECTED
- Strict path validation (Codex MEDIUM): `_validate_job_id` parses with `uuid.UUID`; `validate_source_ext` allowlists media extensions and rejects path-traversal characters; `validate_summary_kind` uses the `SummaryKind` literal
- `atomic_write_*` and the cancel/rmtree path are wrapped in `retry_windows` to handle Windows `PermissionError` (Codex MEDIUM / Gemini HIGH)
- The `POST /jobs/{id}/cancel` route calls `cancel_job`; the `POST /jobs/{id}/stage` route calls `update_stage`; the `POST /jobs/{id}/stale-check` admin/test route calls `mark_stale`
- All new file paths in `data/jobs/<id>/` go through helpers in `app/storage/fs.py`; no other module constructs a `Path` into a job folder by string concatenation

This plan is the file-as-truth contract that the rest of the project relies on. Phase 4 (orchestrator) and every stage adapter call these helpers and never touch `Path("data/jobs")` directly.

## Artifacts this phase produces

### New functions in `app/storage/fs.py`
- `transcript_path(settings, job_id) -> Path` — returns `job_dir(settings, job_id) / "transcript.json"`
- `diarization_path(settings, job_id) -> Path` — returns `job_dir(settings, job_id) / "diarization.json"`
- `summary_path(settings, job_id, kind) -> Path` — returns `job_dir(settings, job_id) / f"summary-{kind}.json"`; first calls `validate_summary_kind(kind)` (raises `ValueError` on bad kind)
- `edits_path(settings, job_id) -> Path` — returns `job_dir(settings, job_id) / "edits.json"`
- `source_path(settings, job_id, ext) -> Path` — returns `job_dir(settings, job_id) / f"source.{validate_source_ext(ext)}"` (validated ext, no leading dot)
- `validate_source_ext(ext: str) -> str` — strips a leading `.`, allowlists `{"mp4","mkv","webm","mov","mp3","wav","m4a","flac","ogg"}`, rejects `..`, `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`, and empty string; returns the normalized (no leading dot) lowercase extension
- `def list_stage_files(settings, job_id) -> list[Path]` — returns the absolute paths of all stage files that currently exist for the job (manifest + transcript + diarization + edits + source.* + summary-*.*)
- `def last_stage_mtime(settings, job_id) -> float | None` — returns the maximum mtime across all stage files in the job folder, or `None` if no files exist (used by the staleness check)

### New functions in `app/jobs/manifest.py`
- `async def read_manifest(settings, job_id) -> JobManifest` — reads the manifest file, validates via `JobManifest.model_validate_json`, returns the model. Raises `FileNotFoundError` (a known error type) if the file is missing, with a message naming the job id
- `def manifest_mtime(settings, job_id) -> float | None` — returns the manifest file's mtime, or `None` on `FileNotFoundError`
- `async def update_stage(settings, session, job_id, stage, manifest_patch: ManifestPatch | None = None) -> JobManifest` — enforces write-manifest-first, commit-DB-last:
  1. Load current manifest via `read_manifest` (raises `FileNotFoundError` → 404 in route)
  2. Build `new_manifest = current.model_copy(deep=True)`
  3. Set `new_manifest.current_stage = stage`
  4. Set `new_manifest.stage_timestamps[stage] = utcnow_iso()` (use `model_copy` to keep the model immutable; if the field is not in the type, use `setattr` on a copy)
  5. Apply the allowlisted `manifest_patch` (if provided) via `new_manifest = new_manifest.model_copy(update=manifest_patch.model_dump(exclude_unset=True))`. The protected fields (`current_stage`, `job_id`, `schema_version`, `stage_timestamps`, `status`, `error`) are NEVER taken from the patch — they are owned by this helper
  6. Write the manifest atomically via `write_manifest` (uses `atomic_write_json` which uses `retry_windows` for the `os.replace` step)
  7. UPDATE the DB row: `current_stage`, `stage_timestamps_json` (JSON dump of `new_manifest.stage_timestamps.model_dump()`), `updated_at`; commit
  8. Return `new_manifest`

### New module `app/jobs/reconcile.py` (NEW — startup reconciliation per Codex HIGH #1)
- `async def reconcile_all(settings, session) -> dict` — walks `data/jobs/<id>/manifest.json` for every `<id>` directory under `data/jobs/`, reads each manifest, and UPDATEs the `jobs` row to match the manifest's `current_stage` and `stage_timestamps_json`. Returns `{"scanned": N, "updated": M, "missing_manifests": [...]}` for logging. The manifest is the source of truth; the DB row is the projection (D-03). Called from the lifespan in `app/main.py` AFTER migrations and BEFORE the app starts serving requests

### New module `app/jobs/resume.py`
- `StageName = Literal["ingested", "transcribed", "diarized", "summarized", "done"]`
- `STAGE_ORDER: tuple[StageName, ...] = ("ingested", "transcribed", "diarized", "summarized", "done")` — the standard walk order from D-12
- `def is_stage_applicable(stage: StageName, manifest: JobManifest) -> bool` — `diarized` is NOT applicable if `manifest.diarization_enabled == False`; `summarized` is NOT applicable if `manifest.summary_kinds == []`; all other stages are always applicable. (Defines the `diarization_enabled` field on `JobManifest` with default `False` — Phase 1 default; Phase 7 flips it to True when the user opts in.)
- `def is_stage_complete(stage: StageName, settings, job_id, manifest: JobManifest) -> bool` — for each stage, check the file-existence rule from D-11:
  - `ingested`: `source_path(...)` exists
  - `transcribed`: `transcript_path(...)` exists
  - `diarized`: `diarization_path(...)` exists
  - `summarized`: for every kind in `manifest.summary_kinds`, `summary_path(..., kind)` exists
  - `done`: returns True iff (a) every applicable prior stage is complete AND (b) `manifest.current_stage == "done"`. The resume rule does not need a `done.json` file
- `def infer_resume_point(settings, job_id, manifest: JobManifest) -> StageName | None` — walks `STAGE_ORDER`: `for stage in STAGE_ORDER: if not is_stage_applicable(stage, m): continue; if not is_stage_complete(stage, ...): return stage`; if the walk completes (every applicable stage is complete), returns `None`
- `def parse_stage_file(path: Path) -> bool` — opens the file, attempts to parse it as JSON (or the appropriate format for source files — for `source.*` always returns True if the file exists and is non-empty); returns True on success, False on any error. Used by D-12 to mark unparseable files for re-run

### New module `app/jobs/cleanup.py`
- `async def cancel_job(session, settings, job_id) -> bool` — ORDERING per Codex HIGH #8: (1) UPDATE jobs SET status='cancelled', updated_at=:now WHERE id=:id; commit. (2) THEN call `retry_windows(shutil.rmtree, job_dir(settings, job_id), attempts=3, backoff_s=0.2, retriable_exceptions=(PermissionError, OSError), ignore_errors=True)`. Reasoning: if step 1 succeeds and step 2 fails, the row is cancelled and the folder can be cleaned up by a future call. If step 1 fails, the folder is untouched and the request returns 5xx. The opposite order (rmtree first, DB second) can leak cancelled rows with no folder. Return True if the row was found, False if no such job
- `async def mark_failed(session, job_id, error: str) -> bool` — UPDATE jobs SET status='failed', error=:error, updated_at=:now WHERE id=:id; commit; return True/False on rowcount
- `def is_stale(settings, job_id, threshold_s: int = 600) -> bool` — get `last_stage_mtime(settings, job_id)`; if None, fall back to `manifest_mtime(settings, job_id)`; return `time.time() - mtime > threshold_s`
- `async def mark_stale(session, settings, job_id, threshold_s: int = 600) -> tuple[bool, bool]` — returns `(stale, marked)` — stale is the is_stale result, marked is whether `mark_failed` actually flipped the row. If stale and not already failed/cancelled, call `mark_failed(session, job_id, "stalled")`
