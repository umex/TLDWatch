# Plan 01-04 — Gap closure: 5 HIGH + 3 MEDIUM follow-ups from Codex implementation review

## Objective

Fix the contract defects Codex flagged in the implementation review (`01-IMPLEMENTATION-REVIEW.md`, risk: HIGH) so Phase 1 ships a foundation whose contracts are truthful in real execution, not just in unit tests. After this plan:

- **H1 (restart-only settings)** — `PATCH /settings` does NOT swap `_State.settings` in memory when the change is a `data_dir` change. The new value is persisted to a `pending` slot in the on-disk JSON (sibling key to `data_dir`) and the in-memory value is left at the boot value. A response header `X-Restart-Required: true` continues to signal that a restart is needed. On boot, the lifespan reads the pending slot, applies it to the in-memory state, and clears the slot. Chained restarts work correctly: each PATCH writes a fresh `pending` slot; the next boot applies it and drops the slot; the subsequent PATCH sees no `pending` slot and behaves like the first.
- **H2 (POST /jobs 201 response)** — The `responses={201: {"model": JobManifest}}` block on `POST /jobs` is removed. `JobManifest` is added to `_EXTRA_OPENAPI_MODELS` in `app/main.py` so it is registered in `components.schemas` (the same path `Transcript` / `Summary` use). The 201 response model is now `JobResponse` (the actual return type).
- **H3 (status projection)** — `update_stage` writes `status` in the same SQL UPDATE as `current_stage` / `stage_timestamps_json` / `updated_at`. The stage-to-status mapping is a single helper `stage_to_status(stage, manifest) -> str` defined in `app/jobs/manifest.py` and called from `update_stage` and `reconcile_all`. Mapping table (see `Stage-to-Status Mapping Table` below).
- **H4 (manifest patch projection)** — `update_stage` SQL extends to UPDATE `language`, `duration_s`, `source_sha256`, `source_type`, `source_path`, `summary_kinds_json` whenever the corresponding field is set in the new manifest. `reconcile_all` SQL extends to project the same columns on boot (currently only `current_stage` + `stage_timestamps_json` are projected).
- **H5 (`create_job` orphan-row compensation)** — `create_job` wraps the folder-creation + manifest-write step in a try/except; on failure the DB row is DELETED and the exception re-raised. A partial orphan (DB row but no folder or no manifest) is now impossible.
- **M1 (Pydantic-validate stage files)** — `parse_stage_file` validates `transcript.json` against `Transcript`, `diarization.json` against an inlined `Diarization` model, and `summary-<kind>.json` against `Summary`. Unparseable / schema-invalid files count as "stage not complete" so the resume rule re-runs them.
- **M2 (zero-byte source rejection)** — `parse_stage_file` returns False for zero-byte `source.*` files (the `source.*` branch gets the same size check as the JSON branch).
- **M3 (status-aware stale check)** — `mark_stale` first SELECTs `status FROM jobs WHERE id = :id`; if status is terminal (`done`, `failed`, `cancelled`), it bails with `(False, False)`. The status check is a single SELECT before the staleness mtime comparison.

Plus the test additions from `STATE.md` "Test additions to plan" — WAL on two connections, migration triple-apply idempotency, `data_dir` PATCH restart-only, `data_dir` validation, OpenAPI 201 response, stage-to-status matrix, manifest patch projection, resume on `{}` / corrupt / zero-byte, stale on terminal rows.

The 78 existing tests must continue to pass (with the small modifications listed below — `test_resume` adds corrupt / `{}` / zero-byte cases; `test_cleanup` adds stale-on-terminal cases; `test_manifest_patch` adds projection tests; `test_create_job` adds orphan-compensation tests).

### Stage-to-Status Mapping Table

The `stage_to_status` helper in `app/jobs/manifest.py` is the single source of truth for stage-to-status projection. The mapping is exhaustive over the five `StageName` literals and the two manifest fields that affect it (`diarization_enabled`, `summary_kinds`).

| `current_stage` value (post-update) | `diarization_enabled` | `summary_kinds` | Computed `status` |
|-------------------------------------|-----------------------|-----------------|-------------------|
| `None` (no stage yet)               | any                   | any             | `queued`          |
| `"ingested"`                        | any                   | any             | `ingesting`       |
| `"transcribed"`                     | any                   | any             | `transcribing`    |
| `"diarized"`                        | `True`                | any             | `diarizing`       |
| `"summarized"`                      | any                   | non-empty       | `summarizing`     |
| `"done"`                            | any                   | any             | `done`            |
| terminal (set by `mark_failed` / `mark_stale` / `cancel_job`) | any | any | `failed` / `cancelled` (NOT touched by `update_stage`) |

The actual `status` value is computed as `_STAGE_STATUS_MAP[stage]`, with the special-cases for `diarized` (only valid if `diarization_enabled`) and `summarized` (only valid if `summary_kinds` non-empty). The terminal statuses (`done`, `failed`, `cancelled`) are not derived from `current_stage`; they are set by `update_stage` (for `done`) and by `mark_failed` / `cancel_job` (for `failed` / `cancelled`). The `stage_to_status` helper handles only the "active processing" mapping; the `reconcile_all` healing pass uses the same helper.

### Modified `app/jobs/reconcile.py::reconcile_all`

- The `updated_at` is the LATEST non-None `stage_timestamps` value (via a helper `_latest_ts(stage_timestamps) -> str` in `app/jobs/manifest.py`); the queued timestamp is the fallback

### Modified `app/settings/service.py`
- Add `class _State`: existing `settings`, `path`, plus `pending: Settings | None = None`
- `apply_update(patch)` no longer updates `_State.settings` for the `data_dir`-changing case. The new flow: build `new`, detect `restart_required`, write the disk dict (which now carries `pending`); if `restart_required`, set `_State.pending = new` AND leave `_State.settings` at the current value. If not `restart_required`, swap `_State.settings = new` (the existing flow; current behaviour for non-`data_dir` patches, which are a Phase 1 no-op)
- New function `apply_pending() -> bool`: if `_State.pending is not None`, set `_State.settings = _State.pending`, set `_State.pending = None`, return True; else return False. Called by the lifespan on boot AFTER loading the disk file
- The on-disk JSON now carries an optional `pending` key next to `data_dir` ONLY when there is a pending change. The `Settings` Pydantic model does NOT include `pending` (it is orthogonal); the persistence layer is a small dict wrapper `{data_dir: str, pending: dict | None}` written by the service via new helpers `_read_disk_dict(path) -> dict` and `_write_disk_dict(path, dict) -> None`. On load, the service returns `(active_settings, pending_settings_or_none)`; the lifespan applies the pending one via `apply_pending`. The `Settings` Pydantic model itself is unchanged
- A module-level constant `_PENDING_KEY = "pending"` is the only string literal
