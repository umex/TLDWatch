Close UAT gap (05-UAT.md test 6, entry 6 — MAJOR; debug session .planning/debug/active-card-silence.md): 05-06's race-closure fix was built on a FALSE premise. 05-06 assumed a late-connecting card's WS connect snapshot carries `status:"starting"` during the model-load + first-chunk wait (ActiveJobCard.tsx:116-119 comment). It does NOT. In the live runtime the DB status for the ENTIRE model-load + transcription window is `"ingesting"`, never `"starting"` and never `"transcribing"`:

- The only writer of DB `starting` is `queue.py:118` (pull_next atomic claim), overwritten milliseconds later by `orchestrator.py:230-233` `update_stage("ingested")` → `stage_to_status("ingested") = "ingesting"` (`manifest.py:41,67-77`).
- `transcribed → transcribing` is persisted ONLY at `orchestrator.py:290-297`, AFTER `transcript = await future` (`:282`) returns, immediately followed by `update_stage("done")` (`:299`). So `transcribing` is never the snapshot status DURING transcription.
- `stage_changed(preparing)` (`orchestrator.py:260`) and `stage_changed(transcribing)` (`:263`) are WS-ONLY with no `update_stage` (comment `:246-250`). A card connecting after they were emitted misses them; `routes_ws.py:186` sends `snapshot.status = job.status` (raw DB = `"ingesting"`).

FE consequence once the card receives `snapshot{status:"ingesting"}`: `isIngesting=true` (ActiveJobCard.tsx:108); `isTranscribingActive` (:121-127) is gated by `!isIngesting` → false, even after progress events set `progressArrived.current`; `isPreparing`'s `status==="starting"` branch (:136) is dead code for this case. The card renders a frozen `"Ingesting File... 0%"` (model-load, no progress.json) / mislabeled `"Ingesting File... X%"` (mid-transcription) — perceived as "nothing is going on" until the job finishes and pops into history. Reproduces with FRESH servers + browser hard reload; the stale-runtime angle was adversarially refuted (Vite serves source on demand keyed by mtime; BE PID 20596 includes 05-05's `stage_changed(preparing)` emission). NOT operational — a code change is required.

The DropZone race makes late-connect the COMMON case: `onJobCreated` fires only when `upload.status==="done"` (`DropZone.tsx:84-87`), by which point the worker has usually already claimed the job and emitted the WS-only `stage_changed(preparing)`.

Fix (Option A — FE-only, cheaper, mirrors 05-06 style): treat the snapshot's `status:"ingesting"` as the preparing/transcribing signal for local ingest (local ingest is instant, so `ingesting` post-snapshot = waiting for model load / first chunk; progress events on top of `ingesting` = transcribing). In ActiveJobCard.tsx:

1. Extend `isPreparing` to also cover `status==="ingesting" && !progressArrived.current` (model-load window for a late-connecting card).
2. Extend `isTranscribingActive` to also fire when `status==="ingesting" && progressArrived.current` (chunks flowing for a late-connecting card) — drop the `!isIngesting` gate so ingesting+progress => transcribing.
3. In the snapshot handler (lines 60-64), also consume `event.stage` (currently ignored) as a fallback signal and read `snapshot.percent` (already sent at routes_ws.py:184) to seed `progressArrived.current` + drive the determinate bar immediately on reconnect so a reconnecting card shows the real percent without waiting for the next progress event.
4. Reorder the label render so `isPreparing` and `isTranscribingActive` take priority over the `isIngesting` "Ingesting File..." label (which is now dead for local ingest post-snapshot), and drop the `!isIngesting` guard from `showIndeterminateBar` so the indeterminate preparing bar renders during the ingesting model-load window.

Rejected alternative (Option B — BE-persisted transient status): persist a transient `preparing`/`transcribing` status to the DB alongside the WS-only `stage_changed` publish in `orchestrator.py` around `:260/:263` so late-connecting snapshots carry an unambiguous `transcribing`/`preparing` status. Heavier and riskier for no additional user-visible benefit over the FE-only fix, which reinterprets already-trusted snapshot fields (status + percent + stage) that routes_ws.py already sends. Rejected for this plan; documented here as the alternative of record.
