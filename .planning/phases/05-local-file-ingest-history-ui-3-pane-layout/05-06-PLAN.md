<objective>
Close UAT gap (05-UAT.md test 5, entry 4 — MAJOR): the 05-05 preparing closure emits an additive WS-only `stage_changed(preparing)` that is NOT persisted to DB/manifest (H3+H4 invariant). The WS connect snapshot (routes_ws.py:167-188) is sourced from `job.status` + `manifest.current_stage` + `progress.json` — none of which carry `preparing`. A card that subscribes AFTER the preparing event was broadcast (common: idle worker picks the job up fast + React 18 StrictMode mount→unmount→remount delays the real socket) receives `snapshot{status:"starting", stage:null}` and misses preparing. Today `isQueued` includes `"starting"` (ActiveJobCard.tsx:107-110), so the snapshot renders "In Queue" with NO bar (`showBar` excludes `isQueued`). The same race affects the transcribing stage: if the card connects after `stage_changed(transcribing)` too, `status` stays "starting", `isTranscribing` is never true, and the card shows "In Queue" for the ENTIRE transcription.

Fix: make the snapshot authoritative for current state on the FE (no BE change — preserves the 05-05 WS-only preparing invariant). Treat DB status `"starting"` as preparing (remove from `isQueued`, add to `isPreparing`) so a late-connecting card shows "Preparing..." + indeterminate bar from the snapshot alone. Add an `isTranscribingActive` branch keyed on `progressArrived.current` (progress events flowing) so a card that missed `stage_changed(transcribing)` but is receiving progress events shows "Transcribing... X%" with a determinate bar derived from `percent`, not from `status === "transcribing"`.

Purpose: The user sees accurate progress feedback regardless of WS connect timing — the "nothing is going on" complaint (test 4/test 5) stops recurring in the common idle-worker case.
Output: ActiveJobCard.tsx snapshot-authoritative state derivation + two race-branch tests covering UAT scenarios (a) and (b). FE-only — no routes_ws.py / orchestrator.py / progress.py changes.
</objective>
