## Phase 5 Interaction Contract

### 1. Drop Zone & Full-Window Drag Overlay
*   **Trigger Mechanism:** A global listener intercepts the window `dragenter` and `dragover` events, preventing browser defaults. When a file drag enters any portion of the browser frame, a full-screen semi-transparent `#FFFFFF` overlay shifts instantly from opacity 0 to active visibility.
*   **Overlay Design:** Features a centered `#2563EB` dashed outer border (with `sm` gap, `md` stroke width) and displays the clear instructional copy: "Drop files to start transcribing". The overlay dismisses cleanly upon `dragleave` or `drop`.
*   **Multi-File Drops:** The dropped items list is captured inside a single-concurrency FIFO client-side queue. Files are uploaded sequentially to keep client memory profile stable.
*   **Idempotency & Re-drops:** For each drop event, the frontend generates a deterministic `Idempotency-Key` header calculated from the file name, byte size, and last modification date (`[filename]-[size]-[lastmodified]`). If an identical file is dropped while an active upload/job runs, the back-end response redirects the UI to collapse onto the existing in-progress job card rather than spawning a duplicate active card.

### 2. Active-Job Card
*   **Layout Placement:** Appears in a vertical list container placed immediately below the primary drop area on `/` and above the completed history list.
*   **Lifecycle States & UI States:**
    *   `queued`: Shows badge with gray label "In Queue".
    *   `ingesting`: Displays progress bar with progress text "Ingesting File... X%".
    *   `transcribing`: Displays a detailed active progress bar displaying "Transcribing... X% (ETA: MM:SS)".
    *   `done`, `failed`, `cancelled`: Terminal states.
*   **WebSocket Event Mapping:**
    *   Upon active-job rendering, the card initializes a WebSocket connection to `GET /ws/jobs/{id}/events`.
    *   On handshake, it parses the `{type: "snapshot"}` event to seed internal state with the current phase, progress value, and initial execution statistics.
    *   The card responds to live incoming `progress` events, shifting percentages. The ETA readout is hidden until the progress tracker collects statistical run times from at least ≥2 processed chunks to prevent jarring ETA calculations.
*   **Terminal State Transition:** When receiving `done`, `failed`, or `cancelled` events, the client plays a brief fade-out transition, destroys the active card's WebSocket connection, and immediately prepends the record into the completed history list below.

### 3. Scroll-Spy Active-Line Highlight
*   **Tracking Mechanism:** Implemented using a single unified `IntersectionObserver` configured with container context.
*   **Focal Anchor:** Set at exactly `50%` vertical height (the direct center of the screen). This center focal anchor reduces eye fatigue by keeping the reading row fixed in the comfortable center focal sweep.
*   **Scroll Calculation:** The container tracks the intersection elements passing through a `rootMargin: "-49% 0px -49% 0px"` threshold (creating a virtual 2% focal line at the center of the container viewport). If rapid user-directed scrolling leaves no single segment actively intersecting, a fallback algorithm computes and highlights the transcript row closest in raw pixel offset coordinates to the absolute viewport center.
*   **Highlight Styling:** The active transcript row displays a solid `4px` left accent border styled with `#2563EB` and a subtle background tint of `rgba(37, 99, 235, 0.05)`.

### 4. Transcript Row Layout
*   **Column Setup:** Uses a crisp CSS Grid row structure containing:
    *   A left-aligned time-stamp panel of fixed `64px` width, showing the formatted text as `[mm:ss]`.
    *   An empty, reserved gutter panel of `80px` width designed to accommodate future speaker labels without destabilizing column alignment.
    *   The main transcript segment body.
*   **Line Height Specification:** Set to exactly `1.5` (representing approximately `21px` line heights) to maximize spatial separation and prevent overlapping visual triggers during scroll-spy boundaries.

### 5. Navigation & Routes
*   **Application Routes:**
    *   `/` (History Index): Shows the file-drop target, active-job panel, and historical transcription rows, where each completed history row renders filename, date, and duration, ordered newest-first.
    *   `/jobs/:id` (Detail View): Hosts the side-by-side split layout showing the Transcript (Left) and Summary placeholder (Right) for a selected job.
*   **Click-Through & Back Actions:**
    *   Clicking a completed row in the `/` history list fires a React Router client-side path change to `/jobs/:id`.
    *   The detail page header features a permanent "← Back to History" button styled as a low-prominence system button. Clicking this executes a client-side route change back to `/`, restoring local list state.

### 6. Loading, Empty, and Error States per Surface
*   **History Empty:** If `/` has zero finished jobs, the history section renders a structured empty layout showing the copywriting template: "No Transcripts Yet".
*   **Transcript Not Ready (404/Processing State):** If a user navigates directly to `/jobs/:id` while the job is transcribing, the API request to fetch the transcript returns a 404. The client intercepts the error, bypasses standard 404 views, and renders an active "Transcribing..." loading animation showing live execution updates retrieved from the companion WebSocket.
*   **Summary Pane Placeholder:** The right panel remains fixed at a stable `40%` width. It displays a centered, low-contrast placeholder text: "Summaries will appear here once summarization is enabled." No dynamic actions occur here.
*   **Layout Stability Stub:** An "Export (Coming Soon)" button is rendered as a disabled grey block inside the detail header to maintain visual stability for future phases without offering clickable behavior.
*   **Upload/Job Error Recovery:** If file ingest fails, the corresponding active-job card is colored with a soft red border and displays the system copywriting error. A small cooperative retry button is offered alongside a "Dismiss Error" action to clear the error state from the screen.
