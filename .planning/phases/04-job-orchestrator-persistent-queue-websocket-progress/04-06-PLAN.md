<objective>
Plan 04-06 closes WR-04 from 04-VERIFICATION.md: the cooperative `queue.cancel` (implemented, tested, D-06-compliant in 04-02) is NOT wired to any API route. The existing `POST /jobs/{job_id}/cancel` route (`post_cancel` at routes_jobs.py:131-155) calls `cleanup.cancel_job` directly, which rmtrees the folder out from under a running orchestrator — destructive, non-cooperative, non-deterministic.

Purpose: For a RUNNING job, the destructive `cancel_job` rmtrees the folder while the orchestrator is still writing to it. The orchestrator's subsequent `atomic_write_json(transcript.json)` fails (directory gone), the exception handler calls `mark_failed` (which may conflict with the already-cancelled row), and the model is unloaded in the finally block. This breaks SC-4 (the user can cancel a queued OR running job; cancellation is idempotent and partial files are cleaned up deterministically — D-06 cooperative cancel). The cooperative `queue.cancel` (queue.py:176-263) already handles all three states correctly: queued → cancel_job + _work_signal.set; running → set the _running threading.Event, let the orchestrator's JobCancelled path do cancel_job + rmtree (no double-rmtree); terminal → no-op returning the row. It just is not reachable from the API.

Output: Rewire `post_cancel` (routes_jobs.py:131-155) to call `queue.cancel(job_id, session, settings)` and map the returned `{status, id}` dict to a JobResponse. Handle the `{}` → 404 case (job not found) and the terminal-no-op case (a job already terminal returns its row as-is, no error). Three API integration tests in `tests/test_cancel.py` (the home 04-02 used for cancel tests, extended): running-cancel via POST /jobs/{id}/cancel, queued-cancel via the API, and idempotent-terminal-cancel via the API.
</objective>
